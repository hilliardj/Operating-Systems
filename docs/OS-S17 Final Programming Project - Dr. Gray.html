Logistics

The final programming project:

  • Is due at the start of the Final Examination period for the class.
  • Must be named "fsparser" on storm, with no filesystem extension and must be self-executable so as to be invoked as: ./fsparser [args]
  • Must be well-commented.
  • Must handle runtime errors without killing the process.
  • Must compile in order to be graded.
  • Must be your own work. Academic misconduct charges will be pursued for those submitting code composed by others.

Description

You are to write a program that takes filenames for FAT filesystems from the command line. The program name must be named fsparser and be self-executable.

If your program is run without command line arguments, it is to print a usage statement detailing the appropriate way to invoke your program.

Your program can accept multiple command-line arguments, each indicating a FAT filesystem image. For example: 

./fsparser 12meg.img 1Gig.img

where 12meg.img and 1Gig.img are FAT filesystem images. Correct invocation of your code requires at least one argument, but can contain multiple arguments.

Your program is to check the image for a valid root directory. If the root directory structure is valid, your code should output: 

Filesystem "12meg.img" has a valid root directory.
The root folder directory listing is:
... an output of the top-level directory follows ... this comment isn't actually printed.


If the image lacks a valid root directory, your code should output: 

Issues found with the image "12meg.img":
... an enumeration of the issues found with the image follows ... again, this comment isn't printed.


If the user specifies more than one command line argument, your program output should simply stack (i.e., concatenate) the results of the individual results.

Official References

You should reference Microsoft's FAT standard for specifics on the filesystem layout.

You should also reference the FAT Root Folder description.

Testing your input

You can (and should!) certainly generate your own filesystem images for the FAT filesystem. For testing, various images will be made for you to test your program on. Look for these to be placed in your home directories as files with extensions ".img".

Do's and Don'ts

  • Do get started early. This task will take you more than 2 days.
  • Don't use code that you have not authored. Ever.
  • Do test your code on images above and beyond those that have been provided to you. (Your code will certainly be graded using images that you have not seen. How will your code handle -42 directory entries?)
  • Don't read in the entire filesystem into memory. That's unnecessary, rather annoying, and will certainly fail on images that your code will be tested on.
  • Do ask me any questions that you have.
  • Don't open the files as text files.
  • Do handle errors caused by incorrect command lines or corrupt filesystem images.
  • Do thoroughly check the 2 web links referencing the Microsoft specification for the VFAT filesystem.
  • Don't do this task without learning about new tools, such as hexedit or hexdump
